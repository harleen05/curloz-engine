curloz-engine
Minimal Vulkan. Maximum Clarity.

A lightweight, modern Vulkan-based graphics engine written in C++. Designed for simplicity, educational clarity, and rapid prototyping, giving developers direct, readable access to Vulkan's core pipeline without heavy abstractions.

▶ Quick Start ↳ Architecture ⎇ GitHub
// 01 · quick start

Get Running in 2 Commands

Clone the repo, configure once, then build and run. The engine launches a Vulkan-rendered triangle as its default demo.

bash — curloz-engine 
$ git clone https://github.com/curloz123/curloz-engine
$ cd curloz-engine

# Configure build environment (only needed once)
$ xmake f -c
✓ checking platform ... linux
✓ checking compiler ... g++
✓ Vulkan SDK found at /usr/local/vulkan

# Compile & launch
$ xmake && xmake run
ℹ
First-time setup: xmake f -c only needs to be run once. After that, xmake && xmake run is the full workflow.
// 02 · prerequisites

What You Need

Ensure the following tools are installed before attempting to build curloz-engine.

  • ✓ Git — version control git --version
  • ✓ xmake — build system v2.8.0+ — xmake --version
  • ✓ Vulkan SDK — graphics API v1.3+
  • ✓ C++ Compiler — g++ / clang++ on Linux/macOS · MSVC / MinGW on Windows
  • ✓ glslc — GLSL → SPIR-V compiler, included with the Vulkan SDK
⚠
Windows users: The engine is currently Linux-optimised. Windows toolchain (MSVC/MinGW), Vulkan SDK path discovery, and path-separator differences may require manual adjustments. See the Roadmap for the upcoming Windows compatibility feature.

// 04 · renderer module

Renderer

The renderer module wraps the core Vulkan lifecycle — from instance creation through to per-frame presentation. It is split across three focused files.

initializers.cpp

Handles Vulkan object creation in dependency order: instance → physical device selection → logical device → swapchain → image views → render pass → pipeline → framebuffers → command pool/buffers → sync objects.

renderer.cpp

Owns the render loop. Each frame acquires a swapchain image, records the command buffer, submits to the graphics queue, and presents. Manages the vkWaitForFences / vkResetFences per-frame synchronisation pattern.

cleaners.cpp

Reverse-order destruction of every Vulkan object. Ensures no validation layer errors on shutdown. Call order mirrors creation order, inverted.

⚡

Pipeline Creation

Programmable shader stages + fixed-function config assembled into a VkGraphicsPipeline.

🔄

Swapchain Management

Handles swapchain creation, image acquisition, and presentation with proper semaphore signalling.

📦

SPIR-V Loading

Reads compiled .spirv binaries from the assets directory and wraps them into shader modules.

🛡

Validation Layers

Debug builds enable Vulkan Validation Layers for API-usage correctness checking.

Window & Surface

The window module provides a thin abstraction over native platform window APIs, exposes the VkSurfaceKHR, and drives the event polling loop.

// 06 · shaders

Shaders

Shaders are written in GLSL and compiled to SPIR-V using glslc (bundled with the Vulkan SDK). Both source and compiled binaries live in assets/shaders/.

Naming Convention

FileStageNotes
triangle.vertVertexGLSL source
triangle_vert.spirvVertexCompiled binary, loaded at runtime
triangle.fragFragmentGLSL source
triangle_frag.spirvFragmentCompiled binary, loaded at runtime

Manual Compilation

glslc — compile shaders 
# Vertex shader
$ glslc assets/shaders/triangle.vert -o assets/shaders/triangle_vert.spirv

# Fragment shader
$ glslc assets/shaders/triangle.frag -o assets/shaders/triangle_frag.spirv
💡
Upcoming: Automatic shader compilation via an xmake after_build rule is planned — tracked in the Roadmap below. This will eliminate the need to run glslc manually.
// 07 · contributing / roadmap

Open Features

These are the current open contribution opportunities. Each is self-contained and has clear module boundaries.

Windows xmake Compatibility & Cross-Platform Validation
Intermediate ⏱ 6–10 hrs

The engine is currently optimised for Linux. Windows Vulkan development uses different toolchains (MSVC/MinGW), Vulkan SDK discovery paths, and path-separator conventions. Contributors on Windows may encounter build or runtime failures. This feature brings full Windows parity.

xmake.lua src/renderer/initializers.cpp README.md assets/
Automatic Shader Compilation via xmake
Beginner ⏱ 3–5 hrs

Contributors currently run glslc manually before each test cycle. Adding an on_build / after_build rule to xmake.lua will auto-compile any modified .vert or .frag to .spv as part of the normal build, removing a friction point for new contributors.

xmake.lua assets/shaders/
